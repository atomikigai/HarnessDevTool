HarnessDevTool — Arquitectura propuesta

Un núcleo en Rust (harness-core) compartido por todas las superficies (UI desktop, CLI, futura web). Expuesto vía un broker JSON-RPC sobre stdio (harness-app-server) inspirado en Codex. Encima, una shell SvelteKit + Tauri con tres módulos verticales: Agentes (Claude CLI), DB Manager y SSH/SFTP Manager.

Flujo de un request (de la UI a la respuesta final)

Usuario escribe en SvelteKit. El store emite thread.send por JSON-RPC al App Server.
App Server enruta al thread manager y abre un nuevo turn en el thread activo.
El core construye el prompt (orden estricto: dev msg → config → AGENTS.md → env → user).
Llama al provider (Anthropic/OpenAI). Streamea SSE.
Cada chunk → item.delta notification → la UI lo pinta en vivo (markdown incremental).
Si el modelo pide tool call: nativa → sandbox; MCP → cliente MCP; módulo → DB/SSH/Agentes.
El resultado se apendiza al historial sin alterar el prefijo (preserva el prefix cache).
Loop hasta que el modelo emite mensaje final → turn.completed → UI cierra el spinner.
Todo se persiste a events.jsonl. Si la UI se cae, el agente sigue; al reconectar replayea.
Acciones riesgosas pausan con approval.request y esperan allow/deny humano.
