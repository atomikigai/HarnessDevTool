HarnessDevTool — Arquitectura (cleanup tras pivote)

Web UI primario (SvelteKit + adapter-node) hablando HTTP+SSE con un backend Rust (harness-server, Axum). El backend orquesta instancias de claude/codex CLI del host vía PTY, y les expone rails determinísticas mediante un MCP server local (harness-bridge). Tres trabajos en paralelo: orquestación (tasks), módulos verticales (DB, SSH) y auto-mejora (Skills, Curator, Learner).

Flujo de un request (de la UI a la respuesta final)

Usuario escribe en SvelteKit. Click "New task" → POST /api/threads/:id/sessions al harness-server.
Backend valida, crea spawn record, lanza claude hijo con PTY + --mcp-config apuntando al harness-bridge local.
Browser ya tenía abierta GET /api/events (SSE multiplex). Empieza a recibir spawn.output items.
PTY bytes → xterm.js en vivo (WebGL2 renderer, 60fps).
El CLI hijo invoca rails: task.claim, memory.search, spec.read, skills.search...
Rails validan, leen/escriben en disco (events.jsonl, tasks/, memory/, skills/) y devuelven JSON tipado.
Tools sensibles (memory.note, skills.manage) disparan approval.request → SSE → modal en UI.
Humano click "Allow" → respuesta HTTP → backend completa la llamada MCP pendiente → CLI continúa.
Task pasa a pending_verify → scheduler asigna a un QA agent (otro spawn).
QA escribe tests, Rust los corre (shell.exec sandboxed), verdict por exit code → task.verified o verify-fail.
Contract diff (Rust): si drift_minor → arbitrator (call corto); si major → re-plan al orchestrator (cap K=2).
Al cierre del thread, learner observa traces y deja drafts en skills/proposed/ para review humano.
Doc canónica en docs/12-build-plan/, docs/13-agents/, docs/14-memory/. Diagrama actualizado tras cleanup de inconsistencias pre-pivote.
